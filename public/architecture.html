From a wholesaler invoice to loyalty points — a self-hosted engine.

A mobile-first PWA that reads Croatian B2B wholesaler invoices — PDF, phone photo, or e-invoice XML — with no paid AI by default. It validates the Croatian OIB checksum, dedupes against past claims, matches Viessmann SKUs, and credits an append-only points ledger where the balance is always reproducible as Σ(delta). Viessmann admins review every submission and run SPIFF campaigns and rewards. The whole app is fully bilingual — EN / HR.

Deterministic core, optional AI

Everything that decides points is reproducible code, not a model. AI is reserved for one job — turning a photo into text — and even that runs on the user's device first.

Deterministic · reproducible

The core runs on plain code — every point is explainable.

  • Regex invoice parsing — unpdf text and a Croatian-invoice parser pull seller, buyer, invoice number, totals and line items.
  • OIB checksum — Croatian mod-11-10 validation on both seller and buyer OIB; mismatches are flagged, not auto-trusted.
  • SKU matching — each line matched to the Viessmann catalog by KPD code, model substring, then fuzzy fallback.
  • Weighted points — points_per_unit × quantity, plus active SPIFF-campaign multipliers and flat per-unit bonuses.
  • Append-only ledger — approval writes one delta row; the balance is always SUM(delta).
Optional · off by default

AI is an input helper, never the source of truth.

  • On-device OCR — Tesseract.js (hrv + eng) runs in the browser for photos; only the extracted text leaves the device.
  • Claude vision — fallback only — used solely for direct image POSTs from non-PWA clients, and only if ANTHROPIC_API_KEY is set.
  • Off by default — the PWA never reaches the vision path; the demo ran 9/9 of Frane's invoices at zero external cost.
  • Same downstream — OCR and vision feed the exact same deterministic parser, validation and ledger as the PDF and XML paths.
Design Principles

A loyalty engine you can audit line by line — built in-house, cheap to run, and private by default.

01
No paid AI by default
Digital PDFs and XML parse for free; photos OCR on-device. The paid Claude vision path is off unless a key is explicitly set — zero required API keys.
02
OIB-verified, no double-claims
Croatian OIB checksums are validated, the buyer must match the installer, and a DB uniqueness rule on (seller OIB, invoice nr, buyer OIB, total) blocks duplicate submissions.
03
Append-only ledger
Points are never edited in place. Every accrual, redemption, adjustment and reversal is a signed delta row, so the balance is reproducible as Σ(delta) and fully auditable.
04
Built in-house, not a SaaS
We chose to build the engine ourselves rather than license a paid loyalty platform like Open Loyalty — keeping data, points logic and the EU-hosted database fully under Viessmann's control.

Receipt Pipeline · Invoice → Scored Line Items · v1.0

Four ways to read an invoice — none of them need a paid API.

A Croatian installer uploads a wholesaler invoice as a digital PDF, a phone photo, or an e-invoice XML. A four-tier strategy turns each into the same structured ParsedReceipt — seller, buyer, invoice number, dates, totals, and line items — then matches every line to a Viessmann product and scores it. Tiers 1–3 are deterministic, free, and run by default; the paid AI tier is off unless an API key is set and the PWA never reaches it.

01

The input boundary

Three document shapes enter the system, plus the installer's session context. The critical design rule: a phone photo is OCR'd in the browser and only its text crosses the wire — the raw image never hits a serverless function.

The 4.5 MB rule
Why photos are OCR'd client-side

A Vercel serverless function rejects request bodies over 4.5 MB, and a phone photo is routinely 5+ MB. So submit/page.tsx branches on file type: images run through tesseract.js on-device, then it fetches /api/receipts/from-text with a JSON body of just { extractedText, fileName }. PDFs & XML go straight to /api/receipts as multipart. The from-text route caps text at z.string().min(20).max(50_000) — kilobytes, not megabytes.

▤
Digital PDF · recommended
Accounting-software / e-invoice export with a real text layer. Sent multipart to /api/receipts; text pulled by unpdf. Instant, free, exact.
◢
Phone photo
JPEG/PNG/HEIC from the in-app camera (capture="environment"). OCR'd by Tesseract in the browser — nothing but the resulting text leaves the phone.
⟨⟩
e-invoice XML
UBL 2.1 / HR-CIUS / PEPPOL BIS 3.0 structured invoice. Detected by text/xml, application/xml, or a .xml extension and parsed losslessly.
◷
Installer context
Every POST runs requireInstaller() first; the resolved installerId scopes the receipt and feeds the downstream fraud / dedup pipeline.

Deterministic by default · AI only as a guarded fallback

The three free tiers cover every real-world path the PWA exercises. Claude vision exists purely as a safety net for scanned PDFs and scripted direct-image uploads — and it stays dark unless an operator explicitly sets a key.

Tiers 1–3 · Deterministic · Free · Default

No API key, no per-document cost, fully private

  • Tier 1 — unpdf: light PDF text extraction in parsePdfTextLight(); the default for any PDF with a text layer.
  • Tier 2 — fast-xml-parser: structured e-invoices via parseEInvoiceXml() with removeNSPrefix:true for dialect tolerance.
  • Tier 3 — tesseract.js: on-device OCR (hrv+eng), dynamically import()ed so the WASM never ships in the initial bundle.
  • All three converge on parseCroatianInvoiceText() — one regex engine, one output shape.
  • For photos, nothing leaves the phone except the text — POSTed to /api/receipts/from-text.
Tier 4 · AI · Optional · OFF by default

Claude vision — a fallback the prototype never triggers

  • Gated: only runs if (process.env.ANTHROPIC_API_KEY) — unset by default, so the path is dead.
  • Narrow scope: scanned/image-only PDFs (no text layer) and direct image POSTs to /api/receipts.
  • The PWA never hits this — its camera flow OCRs on-device and posts text, bypassing the image endpoint entirely.
  • Uses parseReceiptWithClaude() on model claude-haiku-4-5, returning the same JSON schema as the free tiers.
  • If a PDF has no text and no key is set → a clean 422 asking the user to scan to PDF, never a silent paid call.
04

The shared Croatian invoice parser

One regex/heuristic engine in croatian-invoice-parser.ts serves both the unpdf text path and the Tesseract OCR path. It first normalises all whitespace to single spaces (text.replace(/\s+/g, " ")) so the same patterns work whether the source emitted newlines or spaces.

Gate before parse
Reject anything that isn't a Croatian B2B invoice

Two guards must pass or the function returns null (caller treats it as "couldn't read it"): the flattened text must contain at least one OIB and a "total to pay" line.

if (flat.replace(/\s/g, "").length < 50) return null; if (!FIRST_OIB_RE.test(flat) || !TOTAL_PLATITI_RE.test(flat)) return null;
#
OIB · tax id (11 digits)
/OIB\s*:?\s*(?:HR)?\s*(\d{11})/ — tolerates an optional HR prefix. The first match is the seller; the buyer is found under a Kupac…OIB block, else the first OIB that differs from the seller's.
№
Invoice number · Račun
/Racun:?\s*(\S+(?:\/\S+)?)/i — captures forms like 123/01/1, then strips a trailing Kupac the OCR may have glued on.
▦
Issue date · Datum izdavanja
/Datum\s+izdavanja:?\s*(\d{1,2}\.\d{1,2}\.\d{4})/i → normalised to ISO yyyy-mm-dd. Sibling patterns read dospijeća (due) and isporuke (delivery).
∑
Total · Ukupan iznos za platiti
/Ukupan\s+iznos\s+za\s+platiti[^:]*:?\s*([\d.,]+)/i. Companions capture subtotal (…bez PDV) and PDV (EUR).
⌗
Line items · KPD tail
Between the Iznos (EUR) header and the Ukupan iznos bez PDV footer, a global tail regex pulls each row's KPD code, unit, qty, VA %, unit price & amount; the preceding slice is the description.
Per-line tail · LINE_TAIL_RE

Six fields scraped off the right edge of every row

/(\d{2}\.\d{2}\.\d{2})\s+([A-Za-zÀ-ſ]{1,5})\s+ ([\d.,]+)\s+([\d.,]+)\s+([\d.,]+)\s+([\d.,]+)/g
  • $1 — KPD šifra product code (e.g. 28.21.11)
  • $2 — unit / JM (kom, rol …, 1–5 letters)
  • $3 — quantity · $4 — VAT % · $5 — unit price · $6 — line amount
  • Everything left of the match (this row to the previous one) becomes raw_description, capped at 250 chars.
Croatian number handling

Dot = thousands, comma = decimal

  • Croatian invoices write 1.299,00 to mean 1299.00.
  • The parser preserves the original string (e.g. subtotalMatch[1].trim()) — no lossy float conversion at parse time.
  • Conversion happens only at display: parseFloat(total.replace(/\./g,"").replace(",", ".")) in the result UI.
  • Dates flip dd.mm.yyyy → yyyy-mm-dd with zero-padding via the isoDate() helper.
  • The XML tier never touches this — fast-xml-parser already yields canonical decimal strings.

Viessmann B2B Loyalty · Loyalty Engine · Built in-house

From a verified invoice to points, tiers & rewards.

A purpose-built loyalty core for Croatian installers. Every credit lands in an append-only ledger where the balance is simply Σ(delta) — never an editable counter. Campaigns layer multipliers and per-unit bonuses on top, the engine picks the single best non-stackable campaign per line and enforces a per-installer cap, and four tiers unlock progressively richer rewards. We built this rather than license Open Loyalty — fewer moving parts, full control of the audit trail, zero per-seat fees.

The core abstraction

An append-only ledger. Balance is computed, never stored.

balance(installer) = Σ delta  over all points_ledger rows where installer_id = :id
01 · NEVER MUTATED
Rows are only ever inserted
No UPDATE, no DELETE. Earning, spending, refunding and manual correction are all new rows with a signed delta. The live balance is SELECT COALESCE(SUM(delta),0)::int — the exact query the redemption and accrual paths run.
02 · FOUR REASONS
accrual · redemption · reversal · adjustment
Every row carries a reason. accrual from approved receipts, redemption when a reward is claimed, reversal when a redemption is cancelled, adjustment for an admin correction. The reason set is closed and self-explaining.
03 · FULLY TRACEABLE
Every row points back to its cause
Each entry links its origin: receiptId for accruals, redemptionId for spends and reversals, actorId for admin actions, plus a human note. You can reconstruct any balance at any point in time.
04 · REPRODUCIBLE
Re-run the sum, get the same number
Because the balance is a pure fold over immutable rows, it is deterministic and auditable. A disputed total is never "the counter drifted" — it is always a specific row you can read.

Selection · pickBestCampaign()

Best non-stackable campaign per line

  • Skip any campaign whose productFamily is set and ≠ the line's matched family.
  • Candidate points = round(base × bonusMultiplier/100) + round(bonusFlatPerUnit × qty).
  • Discard if the candidate is not strictly better than base points — no campaign can ever reduce an award.
  • Highest finalPoints wins. Exactly one campaign is attached to the line; the rest are ignored. Bonuses never compound.
  • The chosen campaignId + campaignName are stamped onto the receipt_line_items row for audit.
Cap enforcement · per installer

One submission can't drain the budget

  • The pipeline pre-loads, per campaign, how much bonus this installer already claimed: Σ(points_awarded − points_base) over their approved receipts.
  • If capPerInstaller > 0: remaining = max(0, cap − alreadyClaimed).
  • If the proposed bonus exceeds what's left, it is clamped: finalPoints = base + remaining.
  • If clamping leaves nothing above base, the campaign stops being a candidate for that line.
  • capPerInstaller = 0 means unlimited — no cap math runs.
finalPoints = round(basePoints × bonusMultiplier / 100) + round(bonusFlatPerUnit × qty)  →  clamp to base + max(0, capPerInstaller − alreadyClaimed)

Tier-gating

Locked until you reach the tier

  • Every reward declares tierRequired (default Bronze = open to all).
  • On redeem, the API derives the live tier from the balance and calls meetsTier(currentTier, required).
  • Below the bar → 403 with a clear message: "This reward is for {tier} and above. You're currently {tier}." Nothing is charged.
  • The same ranking powers the catalogue UI, so a locked reward reads as locked before the installer even tries.
Refund-on-cancel · admin route

Cancelling makes the installer whole

  • Admin PATCHes a redemption to cancelled. If it was still requested (not shipped):
  • A reversal row of +pointCost is appended — the points return to the balance, no counter is edited.
  • One unit of inventory is restored to the reward.
  • Already-shipped items are intentionally not auto-reversed — that is a separate returns workflow.
  • The action is mirrored to auditLog as redemption.cancelled with the previous status.
04

Manual adjustment

Admins can correct a balance up or down — goodwill credit, a clawback, a fix for a bad import — but only through the ledger, and only with a reason on record.

reason = "adjustment"
POST /api/admin/installers/[id]/adjust

The admin supplies a signed delta and a mandatory reason (validated, min length 2). The engine appends a single adjustment ledger row carrying the actorId of the admin and the reason as its note. There is no "set balance to X" — corrections are deltas, so the history stays intact and the running sum still reconciles.

✎
Reason required
An empty reason is rejected at the API boundary. Every manual move has a stated cause attached to the row.
▤
Audit-logged
A second record lands in auditLog as action points.adjust with the full payload — who, what delta, why.

Why we built it · not Open Loyalty

A deliberate strategic call. The loyalty logic here is small, exact, and tightly coupled to Croatian invoice verification — so owning it beats integrating and paying for a general-purpose platform.

Build · in-house engine

Small surface, total control

  • The whole engine is one ledger table plus a campaign picker — auditable end to end, no black box.
  • Accrual is welded to OIB-verified, deduped receipts; the loyalty rules and the anti-fraud rules share one pipeline.
  • No per-seat or per-MAU licence — cost is just the Neon Postgres it already runs on.
  • Data stays in our EU Postgres; no second system of record, no sync, no extra DPA.
  • Campaign mechanics (best-of selection, per-installer cap) are exactly what the SEE business asked for — not bent to fit a vendor's model.
Buy · Open Loyalty

Powerful, but more than we need

  • A full platform (segments, omni-channel, its own admin) — most of it unused for a single B2B installer program.
  • Another service to host, upgrade, secure and integrate against its API.
  • Licensing / hosting cost for capability we'd never switch on.
  • Points would live in a second system, complicating the receipt → ledger audit story.
  • Custom rules (cap-per-installer, Croatian SKU families) still need bespoke work on top.
Viessmann B2B Loyalty · Dedupe & Fraud · Deterministic guardrails

No invoice earns points twice.

Points are real money to the business, so the gate between a submitted invoice and a ledger credit is strict and deterministic. The cornerstone is the Croatian OIB: the buyer OIB printed on the invoice must match the installer's own account OIB, so a contractor can't claim a colleague's purchase. On top of that sits a duplicate lookup over an indexed invoice tuple, a battery of risk signals, and a four-way status decision that routes the clean ones through and parks the doubtful ones for a human.

01

OIB validation · the anti-double-claim cornerstone

The Croatian OIB is an 11-digit identifier with an ISO 7064 MOD 11,10 check digit. We validate the checksum on both the seller and buyer OIB, then assert the buyer OIB equals the installer's account OIB — the single rule that stops one installer claiming another's invoice.

isValidOib() · oib.ts

The mod-11-10 algorithm, step by step

  • Normalise: strip everything non-numeric (an optional HR prefix and any spaces fall away). Must be exactly 11 digits, else invalid.
  • Seed: start with remainder = 10.
  • Iterate the first 10 digits: add the digit to remainder; take mod 10; if the result is 0, set it to 10; multiply by 2; take mod 11.
  • Check digit: check = 11 − remainder; if check === 10, set it to 0.
  • Accept only if check equals the 11th digit.
Why it's the cornerstone

Buyer OIB must be the installer's OIB

  • The OIB is the government tax ID — it uniquely binds an invoice's buyer to a legal entity.
  • The installer's OIB is fixed at signup and unique in the database; one account = one OIB.
  • If the parsed buyerOib ≠ the installer's oib, the receipt is rejected outright — not merely flagged.
  • So a contractor physically cannot earn points from a wholesaler invoice issued to a different company. The double-claim path is closed at the identity layer.
remainder = 10
for i in 0..9: remainder = (remainder + digit[i]) % 10; if remainder == 0: remainder = 10; remainder = (remainder * 2) % 11
check = 11 − remainder; if check == 10: check = 0  ⟶  valid ⟺ check == digit[10]

Duplicate detection · the invoice tuple

Before crediting, the pipeline looks for an already-approved receipt with the same four-part fingerprint. A match is persisted as a duplicate row — never silently dropped — so the installer can see the invoice was already submitted.

(wholesalerOib, invoiceNumber, buyerOib, totalCents)
receipts_tuple_idx · non-unique

The same physical invoice always produces the same four values: who issued it, its number, who it was billed to, and its exact total in cents. The pre-flight lookup queries this tuple (scoped to status = 'approved' so only an already-credited receipt counts as "the original"). The index is non-unique on purpose — it accelerates the lookup but does not impose a hard DB constraint, so a duplicate submission can still be stored alongside the original instead of erroring out.

Guardrail principles

Deterministic validation. OIB-verified. No double-claims.

01 · DETERMINISTIC
Same invoice, same verdict
Every check is pure logic over the parsed receipt — checksums, equality, date math, an indexed lookup. No model in the loop deciding to approve; the outcome is reproducible and explainable from the fraudFlags.
02 · OIB-VERIFIED
Identity before points
Both OIBs pass the MOD 11,10 checksum, and the buyer OIB must equal the installer's account OIB. The tax identity is the gate — points never accrue to the wrong company.
03 · NO DOUBLE-CLAIMS
The tuple closes the loop
The (wholesalerOib, invoiceNumber, buyerOib, totalCents) fingerprint catches re-submission of an already-credited invoice and parks it as duplicate with a pointer to the original — visible, auditable, zero points.
04 · HUMAN-IN-THE-LOOP
Edge cases reach a person
When something is merely uncertain — a missing OIB, an unmatched Viessmann line — the receipt is held as needs_review rather than auto-rejected. Automation handles the clear cases; people handle the grey ones.

Idempotent points · the key guarantee

Re-deciding an approved receipt reverses the old accrual first

  • On entry the route reads the current receipt. If status is already approved, it looks up every prior accrual ledger row for that receipt.
  • For each, it inserts a matching reversal row with delta = −old, stamped reversedBy = the original ledger id and a note "Admin decision change to …".
  • Only then does it accrue the new figure — so a reviewer can flip approve → reject → approve repeatedly and the balance never double-counts.
  • Balance is always SUM(delta) over the ledger; nothing is ever updated or deleted in place.
The decision contract

One POST, fully validated, fully audited

  • Zod body: decision: "approved"|"rejected", optional note, optional non-negative integer pointsOverride.
  • Guarded by requireAdmin() — non-admins get 403 before any read.
  • Points default to Σ receiptLineItems.pointsAwarded; the override only replaces that total when supplied.
  • A reject writes status + a zero accrual and still reverses any prior approval — clean, predictable, replayable.
  • The note is optional at the API but can be made mandatory in the UI for reject / override flows.

02

Auth & security

Encrypted stateless cookie sessions, modern password hashing with no native build, role-based access, and two layers of tax-id validation — all hosted in the EU.

Sessions · iron-session

Encrypted, signed, HTTP-only cookies

  • An iron-session cookie viessmann_session carries installerId, email, companyName and role — encrypted with SESSION_SECRET.
  • httpOnly, sameSite: "lax", and secure in production; decryption is memoised per request via React cache().
  • Guards: requireInstaller() throws UNAUTHENTICATED; requireAdmin() throws FORBIDDEN unless role === "admin".
  • No server-side session store to run or scale — the cookie is the session.
Passwords · scrypt

Built-in crypto, salted, constant-time

  • Node's native scryptSync with N=16384, r=8, p=1, a per-password 16-byte salt and a 64-byte key — no bcrypt native module to break Vercel builds.
  • Stored as scrypt$N$r$p$salt$hash; verification re-derives and compares with timingSafeEqual.
  • Role-based separation: the same table backs both installers and admins, gated only by role.
  • EU-hosted (GDPR): app on Vercel, data in Neon Frankfurt — personal data stays in the EU.
  • OIB + VIES validation closes the loop on who can register (see below).
VIES · EU VAT registry
lib/vies.ts · fails open, never blocks signup

At signup the OIB is checked against the EU VIES REST endpoint (Croatia uses the OIB as its VAT id, prefixed HR). The call is length-validated to 11 digits, has a 6-second abort timeout, and returns an authoritative flag — on timeout or 5xx it fails gracefully so VIES being down never blocks registration. The result and timestamp are recorded on the installer and surfaced in the admin profile.

#
OIB shape check
An 11-digit Croatian tax id, unique-indexed on the installer table — the local guard before the network call.
✓
Recorded, not gated
viesValidated + viesCheckedAt give reviewers a trust signal without making a third-party service a hard dependency for onboarding.
